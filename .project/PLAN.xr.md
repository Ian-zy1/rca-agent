## 维度 1：目标清晰度（验收可测性）

⚠️ **建议**

多数日产出是具体文件（`models.py`、`tools/prometheus.py`），可测。但存在不可测/缺测的硬伤：

- **D5「切片跑通→8节点图能跑」**：「能跑」二义——是不报错，还是产出正确 RCA？初学者会以「没崩」当通过，掩盖推理空洞。**改为**：给定场景1 告警，8 节点图产出包含「根因文本 + ≥1 条指标证据 + ≥1 条日志证据」的 RCAReport 对象。
- **D7「Prompt 防幻觉优化」**：「优化」是典型无指标动词。而 002 完成标准明确要求「场景1 报告准确率 >70%」，计划里却没有这一步。**改为**：为 3 场景各预设「标准根因」，命中=根因文本含关键词且置信度方向正确，3 中 ≥2 命中即 >70%。
- **Grafana 链接**：002 要求「报告含 Grafana 快照链接」，PRD Day7 有该步，**PLAN 全文未出现**——完成标准被静默丢弃。补一步：报告生成节点拼接 `{grafana_base}/d/{dashboard}?from=...`。

## 维度 2：任务分解合理性

❌ **问题**

1. **FC 与 8 节点固定图存在架构矛盾，却决定了 D3 交付物形态。** PRD §1.3/§3.2 称「LLM 用 Function Calling 自主决定何时查 Prom/Loki」（agent loop 语义），而 §3.1 是固定 8 节点管线（控制流写死，LLM 不做路由决策）。二者只能活一个：固定图里「指标分析/日志分析」节点若直接 `query_prometheus(写死的promql)`，则 **D3 学的 FC 根本没被用于控制流**，D3 交付物形同虚设；若节点内用 FC 让 LLM 动态生成 PromQL，则必须在 D3 前写明「FC 用于节点内查询生成，不用于节点间路由」。计划把它推迟到「D5 扩展时二选一」，但 D3 的产出（FC 工具集 schema 形态）依赖这个答案。**后果**：周六上午才发现 D3 做错方向，返工吃掉下午扩 8 节点的时间。**必须在 D2 末尾定调**，写进计划。

2. **「切片→扩展，同一份代码长大」是部分失实，隐藏返工。** 3 节点切片是 `receive_alert → investigate → infer_and_report`，扩成 8 节点时 `investigate` 被拆成 5 个、`infer_and_report` 被拆成 2 个——**切片里 2/3 的节点被丢弃重写**。真正复用的只有 State 定义、receive_alert、D1–D4 的工具函数、图骨架。对 0 基础学习者，更省时的做法是直接搭 8 个 stub 节点先验证 LangGraph 管线，再逐个填实现，零丢弃。**后果**：周六上午在切片的 `investigate` 单体逻辑上过度投入（因为「它要长大」），下午全删重来。**改法**：把切片显式定位为「只验证 State+边+编译能跑，investigate 用 3 行 Mock」，禁止往里写真实推理。

3. **对话模式工作流未定义却排进 D6/D8。** 计划「未解决」区自己写了「对话模式工作流（意图识别/慢SQL定位节点）PRD 未定义」，但 D6 下午排了「对话模式」、D8 要演示场景3（慢查询对话）。**排了未定义的工作 = 必滑期。** **改法**：D2 就把对话模式的节点裁剪定死（建议：对话入口→跳过告警接收/聚合→直接进拓扑关联，不做意图识别节点），写进计划，否则砍场景3。

4. **接口数量三套口径未统一就开建。** 002 说 3、PRD §5 列 6、PLAN D6 写「4 接口」。**改法**：D6 前钉死为「POST /alerts、POST /chat、GET /reports/{id}、GET /health」4 个（list/trace 砍掉），写进计划。

## 维度 3：时间估算现实性

❌ **问题**

1. **结构性零缓冲，且用「借 D8」伪装成已解决。** CONTEXT 明确「Oracle 已标 6 个必修项，含**无缓冲日**」。PLAN「已解决」区列了 4 条，没有一条是真正加缓冲——「D7 超载→借 D8」是把 deadline 当天（6/30 演示日）的容量挪给 D7，等于**演示日自己没余量**。D7 极可能滑（见下），D8 同时背「场景3 + 排练×3 + D7 溢出」，演示必抖。**后果**：比赛当场翻车。**改法**：把 D5/D6 某个非关键产出（如 Gradio 美化、对话模式）显式标为「可砍」，腾出 D8 半天做缓冲；或把演示排练提前到 D6 晚。

2. **D3（FC）1.5h 对 0 基础不现实——Oracle 已标，PLAN 只换了天号没解决。** 要在 1.5h 内：学 FC + JSON Schema + tool_call 流程 + 封装 `query_prometheus()` + `query_loki()` + 定义 FC schema。FC 单概念首学通常 2–3h，且 DeepSeek 经硅基流动 OpenAI 兼容端的 `tools` 参数有自身坑。**改法**：D3 只做「学 FC + 1 个工具（query_prometheus）跑通 tool_call」，`query_loki` 推到 D4 上午前 30min；或 D2 压缩拓扑编写，给 D3 让 30min。

3. **D5 上午首次接触 LangGraph，却要 3h（扣 Q&A 后实为 ~2h）出端到端切片。** LangGraph 首学者的典型坑：TypedDict state 合并语义、`add_conditional_edges` 的返回值映射、compile 后 invoke 的入参层级。2h 内从零到端到端可跑，对 0 基础是赌运气。这是**全计划最高风险时段**。**改法**：D4 晚（周五）匀出 30min 过一遍 LangGraph 最简 2 节点 hello-world，把「首次接触」前移，D5 上午才有可能。

4. **D6 上午 3h 要做「真实 Prom/Loki 接入 + RAG 节点 + 报告节点」。** 注意 D3 建的 `query_prometheus/loki` 一直只对 Mock，到 D6 才首次打真实数据——**工具集成 bug 全堆在负载最重的 D6**。且 topology.yaml 里的 PromQL（`histogram_quantile(...)`）与真实 Prometheus 标签匹配是调试黑洞。3h 不够。**改法**：D3/D4 工具封完立刻用真实环境各跑 1 条查询验证（哪怕只 print），把集成风险前移。

## 维度 4：依赖与风险完整性

⚠️ **建议**

依赖链主干正确，但多处「已有真实环境」被当成免验证前提：

- **真实 Prometheus/Loki 标签 vs topology.yaml 命名一致性**——最大隐式依赖。D2 写 `order-service`/`node-01` 等名，若真实 Prom 里标签是 `job=ordersvc`、节点叫 `worker-1`，则所有指标/日志查询返回空，RCA 无证据，场景1 当场穿帮。**改法**：D2 第一步先 `curl` 真实 Prom 拉 `up{}` 与 `node_uname_info` 看真实标签，**据实写 topology.yaml**，不要凭想象命名。计划应把「核对真实标签」列为 D2 首个动作。

- **硅基流动 FC + bge-m3 embedding 端点支持**——PLAN 假设二者可用，但 D1 验收只写「API可用」（chat 端点）。若 FC 的 `tools` 参数或 `/embeddings`（bge-m3）在该平台不稳定，D3/D4 直接塌。**改法**：D1 验收清单加两条——「FC tool_call 返回有效」「embeddings 接口返回向量」。这是 5 分钟的事，却保 D3/D4 两条命。

- **ChromaDB 在 macOS 首装**——chromadb 拉 onnx/tokenizers 等重依赖，darwin 上偶发 20–40min 或编译失败。PLAN D1 就 `pip install chromadb`（好：前置去风险；坏：可能吃掉 D1 的 1.5h）。**改法**：D1 只装 `langchain langgraph openai`，chromadb 单独挪到 D4 开始时装（反正 D4 才用），失败也有整个 D4 周末前缓冲。

- **Alertmanager webhook 对接**——D6 下午 3h 里夹了「FastAPI 4 接口 + Alertmanager webhook + 对话模式」。配置 Alertmanager 向本地 dev 端口 POST 测试告警本身就费时。**改法**：webhook 对接用 `curl` 手发 PRD §5.1 的 JSON 模拟，不真接 Alertmanager（demo 评委也分不出），省 1h。

## 维度 5：隐含假设与遗漏项

❌ **问题**

承重但未写明的假设过多，逐条点名：

1. **假设「1.5h 工作日 = 学新概念 + 建模块 + 5 题 Q&A」三件事能塞同一段时间。** 实际学+建已占满 1.5h（尤其 D3 FC、D4 RAG），Q&A 必然累积为债，「周末上午补欠的」又与 D5/D6 上午的高风险切片/接入抢同一块时间。**后果**：35 题 Q&A 静默破产，理论 30–50% 权重裸奔。**改法**：明确「工作日不做 Q&A，全部集中周末上午 1h 刷」，或把 Q&A 降到「每天 1 题精读」。

2. **假设「切片 = PRD §8 降级预案」。** §8 降级是「LangChain Chain 线性流程」，切片是「LangGraph 3 节点」——**两者不同框架**，复用宣称错误。这不是大问题，但会让作者在「是否投入打磨切片」上误判（以为切片是保底方案，其实保底方案另需一条 LangChain 链）。**改法**：删掉该句，或把 §8 降级预案改为「3 节点 LangGraph 切片」并同步改 PRD。

3. **假设 D3 的 FC 工具会被 8 节点图使用**（见维度2第1条）——取决于未解决的架构选择，是循环依赖。

4. **假设「真实环境」包含场景1(OOM)所需的 PodOOMKilled 告警 + OutOfMemoryError 日志 + 内存指标**。若真实环境没这套数据，场景1 无从演示，而造数据排在 D7（场景2）才出现——**场景1 的 demo 数据准备在计划里缺位**。**改法**：D5 切片用的「场景1 Mock」数据要沉淀成可复用 fixture，D6 真实接入时若真实环境缺数据，直接回退该 fixture 注入。

## 整体评价

这是一份**组织清晰、决策透明、比 PRD §7 原计划显著更务实**的统一日历，合并两套冲突日历、修正星期错位、Interleaved 学建结合都对路。但它**用「借 D8」糊弄了 Oracle 已标红的无缓冲问题**，且把三个**应在 D3 前定调的架构/数据假设**（FC 用途、对话工作流、真实标签匹配）推迟到周末——这些会在 D5/D6 集中爆雷，正好砸在零缓冲的 deadline 路径上。

## 必须修改项（❌）

1. **加真缓冲**：砍掉一项非关键产出（对话模式 / Gradio 美化）腾出 D8 半天，或排练提前；「借 D8」不计缓冲。
2. **D2 末尾定调 FC 用途**（节点内查询生成 vs 节点间路由），写进计划，决定 D3 交付物形态。
3. **D2 定义对话模式工作流**（建议跳过告警接收/聚合直入拓扑，无意图识别节点），否则砍场景3。
4. **D2 首动作核对真实 Prom/Loki 标签**，据实写 topology.yaml；场景1 的 Mock fixture 要可复用为 demo 数据。
5. **接口数量钉死**（建议 4 个：alerts/chat/reports/{id}/health），写入计划。

## 建议改进项（⚠️）

1. D5 验收改可测（产出含根因+证据的 RCAReport）；D7「优化」改为「3 场景 ≥2 命中标准根因」；补 Grafana 链接生成步骤。
2. D3 FC 1.5h 超载——`query_loki` 挪 D4，或 D2 让 30min。
3. D4 晚加 30min LangGraph hello-world，前移「首次接触」风险。
4. D3/D4 工具封完立刻打真实数据各验 1 条，别把集成 bug 全留到 D6。
5. 切片定位收紧为「只验骨架」，禁止写真实推理；删除「切片=§8降级预案」的错误复用宣称。
6. D1 验收加「FC tool_call 可用」「bge-m3 embeddings 可用」两条去风险项。
7. Q&A 改为周末集中刷，或降为每天 1 题，避免沉默债务。
8. chromadb 装包挪到 D4 开始，避免吃掉 D1。

## 亮点

- **双轨合并 + 真实日期对齐**：废止两套冲突 Day 号、用 2026 真实星期纠偏，是 PRD §7 的实质性修正，执行力强。
- **「已采纳决策 / 未解决」分区**：把权衡摆到台面，便于评审与后续追踪，工程素养在线。
- **Gradio 砍 React**：在 0 AI 基础 + 时间紧约束下是正确取舍，避免前端黑洞。
- **依赖链显式列出**：`env→模型→FC→RAG→切片→8节点→API→场景` 清晰，比 PRD §7 的隐式顺序更可控。
- **Interleaved 学建**：对有 6 年开发经验、0 AI 基础的人，学完立刻用是最优路径，好于先学后建。

## 盲区清单（最可能咬人的隐式假设）

1. **真实 Prom/Loki 标签命名 = topology.yaml 命名**——若不一致，指标/日志节点全空，RCA 无证据，当场穿帮（最高危）。
2. **1.5h 工作日能同时容纳「学+建+5题Q&A」**——乘性超载，Q&A 必然违约，理论 30–50% 裸奔。
3. **周六首次学 LangGraph 即可 2–3h 出可跑切片**——LangGraph 状态合并/条件边是初学者高频卡点，2h 内端到端是赌运气。
4. **D3 的 FC 交付物会被 8 节点图消费**——取决于未定调的架构选择，可能整块白做。
5. **「借 D8」= 加了缓冲**——实为挪用演示日容量，D7 一滑即多米诺到 demo 现场。
6. **ChromaDB 在 macOS 首装顺利**——重依赖偶发长编译/失败，可能吃掉整个工作日时段。
