# Part 1 · AI / 大模型常见术语(30 个必知)

## 一、基础概念(7 个)

### 1. LLM (Large Language Model · 大语言模型)
- **是什么**: 用海量文本训练出来的超大型 AI,能"理解"和"生成"人类语言
- **开发者类比**: 一个读过整个互联网的"超级函数",输入文本返回文本
- **例子**: GPT-4、DeepSeek-V3、Qwen(通义千问)、Llama

### 2. 参数量(Parameters · 参数)
- **是什么**: 模型内部"旋钮"的数量,决定模型能力上限
- **开发者类比**: 类似数据库的索引数量,越多越精细但越慢越贵
- **常见**: 7B(70亿)、13B、70B、175B。B = Billion = 10亿
- **读法**: "7B" 读作"七十亿参数"

### 3. Token(词元)
- **是什么**: 大模型处理文本的最小单位。不是"字"也不是"词"
- **规则**: 1 个英文单词 ≈ 1-2 token;1 个汉字 ≈ 1.5-2 token
- **为什么重要**: API 按 token 计费,上下文窗口也按 token 算
- **例子**: "根因分析" ≈ 3-4 token;"Root Cause Analysis" ≈ 4 token

### 4. Context Window(上下文窗口)
- **是什么**: 一次对话能塞进去的最大 token 数
- **开发者类比**: 类似数据库查询的 `LIMIT`,超了就截断或报错
- **常见**: 8K / 32K / 128K / 200K
- **影响**: 窗口越大,能塞的文档越多,但越贵越慢

### 5. Inference(推理)
- **是什么**: 模型训练好之后,接收输入、生成输出的过程
- **开发者类比**: 训练 = 编译;推理 = 运行
- **注意**: "推理"不是逻辑推理,是"运行模型"

### 6. Hallucination(幻觉)
- **是什么**: 模型一本正经地胡说八道,编造不存在的事实
- **开发者类比**: 类似 NullPointerException——看起来正常但结果完全错
- **RCA 中的危险**: 模型可能编造不存在的指标或日志,必须用 RAG 约束

### 7. Multimodal(多模态)
- **是什么**: 模型能同时处理文本、图片、音频、视频等多种输入
- **例子**: GPT-4o 能看图说话
- **RCA 中**: 未来可用于分析 Grafana 截图或日志截图

## 二、Prompt 相关(6 个)

### 8. Prompt(提示词)
- **是什么**: 你给模型的输入文本,决定输出质量
- **开发者类比**: Prompt 是"需求文档",写得好=输出好
- **核心**: 垃圾进垃圾出(Garbage In Garbage Out)

### 9. System Prompt / User / Assistant(三个角色)
- **是什么**: 对话中的三种角色
  - **System**: 设定模型身份("你是 SRE 专家")
  - **User**: 你的问题("分析这个告警")
  - **Assistant**: 模型的回复
- **开发者类比**: System = 环境变量;User = 命令行参数;Assistant = 程序输出

### 10. Temperature(温度)
- **是什么**: 控制输出随机性,0~2 之间
- **规则**:
  - `0` = 每次输出一样(确定性)
  - `1` = 有随机性(有创造力)
  - `2` = 非常随机(可能乱说)
- **RCA 该用多少**: `0`,根因分析要确定性,不能每次结果不同

### 11. Top_p(核采样)
- **是什么**: 和 Temperature 类似,控制采样范围。只从概率前 p% 的词里选
- **规则**: 通常和 Temperature 二选一,不同时调
- **建议**: 调了 Temperature 就别动 Top_p(默认 1)

### 12. Few-shot(少样本提示)
- **是什么**: 在 Prompt 里给几个示例,让模型"照着做"
- **开发者类比**: 给函数写单元测试的输入输出样例
- **对比**: Zero-shot(不给示例)/ One-shot(给一个)/ Few-shot(给多个)

### 13. Chain-of-Thought / CoT(思维链)
- **是什么**: 让模型"一步一步想",而不是直接给答案
- **开发者类比**: Debug 时加 print 看中间过程,而不是只看最终结果
- **用法**: Prompt 里加"请一步步分析"或"Let's think step by step"
- **RCA 必用**: 根因推断必须用 CoT,否则模型可能跳到错误结论

## 三、训练相关(6 个)

### 14. Pretrain(预训练)
- **是什么**: 用海量文本(网页/书籍/代码)训练模型的基础能力
- **开发者类比**: 大学通识教育,什么都学一点
- **产出**: 基座模型(Base Model),如 Qwen-7B-Base
- **特点**: 贵(百万到千万美元级)、耗时(数周到数月)

### 15. SFT (Supervised Fine-Tuning · 指令微调)
- **是什么**: 用"问答对"数据训练模型听指令
- **开发者类比**: 职业培训,学具体技能
- **产出**: 指令模型(Chat Model),如 Qwen-7B-Chat
- **数据格式**: `{"instruction": "...", "input": "...", "output": "..."}`

### 16. RLHF (Reinforcement Learning from Human Feedback · 人类反馈强化学习)
- **是什么**: 让人类给模型输出打分,用强化学习让模型"更讨人喜欢"
- **开发者类比**: 绩效考核,根据反馈调整行为
- **作用**: 让模型回答更安全、更有用、更符合人类偏好
- **现状**: DPO(Direct Preference Optimization)是更简单的替代方案

### 17. Fine-tuning(微调)
- **是什么**: 在已有模型基础上,用你的数据继续训练
- **开发者类比**: Fork 一个开源项目,改成你需要的
- **用途**: 让模型学会特定领域知识或风格
- **注意**: 改的是模型"行为",不是"知识"(知识用 RAG)

### 18. LoRA (Low-Rank Adaptation)
- **是什么**: 一种高效的微调方法,只训练很少的参数(原模型的 0.1%-1%)
- **开发者类比**: 不改主程序,只加一个"补丁包"
- **优势**: 普通 GPU 就能跑(7B 模型 LoRA 只要 8-16GB 显存)
- **用途**: 比赛中"大模型调优"考的就是这个

### 19. Quantization(量化)
- **是什么**: 把模型的参数从高精度(FP16)压缩到低精度(INT8/INT4)
- **开发者类比**: 图片压缩,从 PNG 变 JPEG,文件变小但画质略降
- **效果**: 7B 模型 FP16 要 14GB,INT4 只要 4GB
- **用途**: 让大模型在普通电脑上跑

## 四、应用相关(7 个)

### 20. Embedding(嵌入 / 向量化)
- **是什么**: 把文本变成一串数字(向量),让计算机能算"语义相似度"
- **开发者类比**: 给每段文本生成一个"指纹",相似文本指纹相近
- **例子**: "Pod 内存溢出" → [0.12, -0.34, 0.56, ...](768 维)
- **用途**: 语义搜索、RAG、相似度计算

### 21. Vector Database(向量数据库)
- **是什么**: 专门存储和检索向量的数据库
- **开发者类比**: 类似 Elasticsearch,但搜的是"语义"而不是"关键词"
- **常见**: Chroma / Milvus / Pinecone / FAISS
- **用途**: RAG 的存储层

### 22. RAG (Retrieval-Augmented Generation · 检索增强生成)
- **是什么**: 先从知识库检索相关文档,再喂给 LLM 回答
- **开发者类比**: 开卷考试——先翻书找答案,再用自己的话回答
- **流程**: 提问 → 向量检索 Top-K 文档 → 拼接 Prompt → LLM 生成
- **解决**: 幻觉 + 知识时效 + 私有数据

### 23. Chunking(文档切分)
- **是什么**: 把长文档切成小块,每块单独向量化
- **开发者类比**: 类似数据库分页,一页一页存
- **策略**: 固定大小(512 token)/ 递归切分 / 按语义切分
- **影响**: 切太大检索不准,切太小丢失上下文

### 24. Agent(智能体)
- **是什么**: LLM + 工具 + 自主决策 = 能完成复杂任务的 AI
- **开发者类比**: LLM 是大脑,工具是手脚,Agent 是完整的"人"
- **核心**: 模型自己决定"调什么工具、传什么参数、什么时候停"

### 25. Function Calling / Tool Use(函数调用 / 工具调用)
- **是什么**: 你把函数签名告诉 LLM,LLM 自己决定调不调
- **开发者类比**: 给 LLM 注册一个"API 网关",它自己决定路由
- **流程**: 问模型 → 模型返回 `{"name":"query_prometheus","args":{...}}` → 你执行 → 结果回传模型 → 最终回答
- **RCA 核心**: 让模型查 Prometheus、查 Loki,这就是 Agent 的发动机

### 26. ReAct (Reasoning + Acting)
- **是什么**: Agent 的一种模式,交替"思考"和"行动"
- **流程**: Thought(想) → Action(做) → Observation(看结果) → Thought → ...
- **开发者类比**: Debug 的循环——假设→改代码→运行→看结果→再假设
- **vs Function Calling**: FC 是单步;ReAct 是循环

---

## 五、架构相关(2 个)

### 27. Transformer
- **是什么**: 几乎所有现代大模型(GPT/Qwen/Llama)的底层架构
- **核心**: Self-Attention 机制——让模型理解词与词的关系
- **开发者类比**: 类似一个"全局上下文感知的解析器",不像 RNN 只看前面
- **你只需知道**: 大模型都用这个架构,不用懂数学

### 28. Attention / Self-Attention(注意力机制)
- **是什么**: 让模型在处理一个词时,"关注"到其他相关的词
- **例子**: 处理"苹果"时,attention 会关注上下文判断是水果还是公司
- **开发者类比**: 类似 SQL 的 JOIN——根据关联性把信息拼起来

---

## 六、工程/框架相关(2 个)

### 29. MCP (Model Context Protocol · 模型上下文协议)
- **是什么**: Anthropic 提出的标准协议,让 AI 安全地访问外部工具和数据
- **开发者类比**: 类似 OpenAPI/Swagger,但是给 AI Agent 用的
- **用途**: 标准化"AI 调工具"的方式,阿里 sysom_mcp / ack-mcp-server 都用这个

### 30. Prompt Engineering(提示工程)
- **是什么**: 系统性地设计和优化 Prompt,以获得更好的输出
- **开发者类比**: 类似 SQL 调优——同样的数据,查询写法不同性能差很多
- **核心技巧**: 角色设定 / Few-shot / CoT / 结构化输出约束

# Part 2 · UnifiedModel incident-investigation 是什么

## 一句话回答

**阿里官方开源的一个完整可运行的 RCA(根因分析)Agent demo**,演示了"如何用 AI 智能体自动排查微服务故障"。

## 它演示了什么场景

```
故障: 支付网关 SLO 违规(延迟超标)
   ↓
Agent 自主执行:
   1. 发现支付网关延迟异常
   2. 沿拓扑图向上游追溯(跨域遍历)
   3. 按 Runbook 诊断协议逐步排查
   4. 定位根因: checkout-service 发起了"重试风暴"(大量重试导致雪崩)
   5. 输出根因报告
```

**通俗理解**: 就像一个"SRE 机器人",接到告警后自己查指标、查日志、查拓扑,最后告诉你"哪里坏了、为什么坏了"。

## 它的组成(文件结构)

```
examples/incident-investigation/
├── start.sh              ← 一键启动脚本
├── seed-data/            ← 种子数据
│   ├── prometheus/       ← Prometheus 模拟指标数据
│   └── elasticsearch/    ← ES 模拟日志数据
├── umodel-config/        ← UModel 配置(实体/关系/拓扑定义)
│   ├── entities.yaml     ← 实体定义(服务/Pod/Node)
│   ├── telemetry.yaml    ← 遥测数据定义(指标/日志)
│   └── topology.yaml     ← 拓扑关系(谁依赖谁)
├── runbooks/             ← 诊断手册(结构化专家经验)
│   └── latency-runbook.yaml  ← 延迟故障的诊断流程
└── README.md             ← 说明文档
```

## 它用了哪些技术(架构)

```
┌──────────────────────────────────────────┐
│            LLM (大模型推理)               │
│     (通义千问 / 任何 OpenAI 兼容 API)     │
└──────────────────┬───────────────────────┘
                   │ 调用
┌──────────────────▼───────────────────────┐
│          UModel (语义层 / 本体论)         │
│  ┌─────────┐  ┌──────────┐  ┌─────────┐ │
│  │ EntitySet│  │Telemetry │  │ Storage │ │
│  │ (实体)   │  │DataSet   │  │ (存储)  │ │
│  └────┬────┘  └────┬─────┘  └────┬────┘ │
│       └────── Link(关系) ────────┘      │
└──────────────────┬───────────────────────┘
                   │ 通过 MCP 协议查询
    ┌──────────────┼──────────────┐
    ▼              ▼              ▼
┌───────┐   ┌──────────┐   ┌──────────┐
│PromQL │   │ES/Loki   │   │ CMDB     │
│查指标  │   │查日志    │   │查拓扑    │
└───────┘   └──────────┘   └──────────┘
```

**通俗解释**:
- **LLM** 是大脑,负责推理
- **UModel** 是"翻译层",让大脑能看懂运维数据(知道"这个指标属于哪个服务")
- **MCP 工具** 是手脚,负责实际查 Prometheus / ES / CMDB

## Runbook 是什么(关键概念)

Runbook = **结构化的专家诊断经验**。不是纯文本,而是拆成 5 类:

| 类型 | 作用 | 例子 |
|---|---|---|
| Observation(观察) | 该看什么指标 | "查支付网关 P99 延迟" |
| Toolkit(工具) | 用什么工具查 | "用 PromQL 查 http_latency" |
| Knowledge(知识) | 判断标准 | "P99 > 500ms = 异常" |
| Automation(自动化) | 能自动做什么 | "自动拉取上下游链路" |
| Skill(技能) | 复合能力 | "重试风暴识别" |

**对比纯 RAG**: RAG 是"把文档扔进去让模型自己找";Runbook 是"给模型一份结构化的排查清单",精准得多。

## 怎么跑起来

```bash
# 1. Clone 仓库
git clone https://github.com/alibaba/UnifiedModel.git
cd UnifiedModel/examples/incident-investigation

# 2. 配置 LLM API Key
export OPENAI_API_KEY="你的key"
export OPENAI_BASE_URL="https://api.siliconflow.cn/v1"  # 或 DashScope

# 3. 一键启动(会自动起 Prometheus + ES + 种子数据)
./start.sh

# 4. 观察输出:Agent 会打印每一步推理过程
```

## 对你的价值(为什么值得看)

| 价值 | 说明 |
|---|---|
| **理解完整 RCA 链路** | 从告警到根因,每一步怎么做的 |
| **学习 UModel 建模思路** | 比你手写 topology.yaml 更系统 |
| **学习 Runbook 设计** | 比纯 RAG 精准的诊断协议 |
| **学习 MCP 工具封装** | 每个诊断动作如何抽象成工具 |
| **比赛加分** | 能说"参考了阿里 UModel 的设计思路" |

## 和你的 RCA 智能体的关系

```
你的计划:                    阿里的实现:
─────────                    ──────────
LangGraph 状态机      ←→     UModel Agent
topology.yaml         ←→     UModel entities/topology
Function Calling      ←→     MCP 工具集
RAG 检索历史案例       ←→     Runbook 知识库
Prometheus 查询       ←→     PromQL MCP 工具
```
