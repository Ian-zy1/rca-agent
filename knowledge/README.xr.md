## 维度一:目标清晰度

### ❌ 问题 — 学习目标几乎全部不可测试

每天"学习目标"使用 `理解`/`会`/`让` 等不可验证动词:

| 位置 | 目标表述 | 问题 |
|---|---|---|
| Day 1 | "理解 LLM 是文本输入/文本输出的函数" | "理解"无法判定达成 |
| Day 2 | "会写结构化 Prompt" | "会写"标准是什么? |
| Day 5 | "理解语义搜索" | 同上 |
| Day 6 | "理解状态机式编排,会用 LangGraph" | "会用"到什么程度? |

验收清单(§三)是**空模板占位符**,实际验收标准未定义。

**不修改的后果**: 7 天学完无法判断"到底学会了没有",比赛前一天才发现某个 Day 没掌握,没有补救时间。

**改法**: 把每个目标改写为可观测行为。例如 Day 4 改为:
> "能独立写出 query_prometheus 工具的 JSON Schema(不查文档),并让模型在给定告警时正确返回 tool_call"

做到没做到一目了然。验收清单在本次评审阶段就填具体项。

## 维度二:逻辑完整性

### ❌ 问题 1 — 理论与实操权重严重失衡(最大风险)

§1.1 明确"考核形式:理论 + 实操",理论含"人工智能基础理论算法、大模型原理与调优、智能体设计"。但 7 天计划**100% 是实操 API 调用**,理论被塞进每天 5 个 Q&A(共 35 题)+ 周末"可选"附录(且标"非必学")。

具体缺口:
- **大模型原理**: Transformer/Attention 被砍到附录"中"优先级,但"大模型原理"是明考项
- **大模型调优**: LoRA/量化标"低"优先级,但"调优"是明考项
- **智能体设计**: LangGraph 只讲"怎么用","什么是好的智能体架构"几乎没覆盖

**不修改的后果**: 理论题丢分。理论通常占 30-50%,对 0 基础选手恰恰是拉分项——你做了 RCA 项目已经很能展示实操,理论反而是短板。

**改法**: 每天增加 1 个"理论卡片"环节(15 分钟),覆盖:
- Day 1-2: Transformer/Attention 一句话原理
- Day 3-4: Pretrain→SFT→RLHF 三阶段
- Day 5-6: 智能体设计原则(ReAct vs Plan-and-Execute)
- Day 7: LoRA/量化/部署 一句话

### ❌ 问题 2 — "大模型部署"实操完全缺失

§1.1 实操明确含"大模型部署"。但 7 天计划没有部署内容(Day 7 是 RAG 串联,不是部署)。

**改法**: Day 7 砍掉一半 RAG 内容,加入"用 FastAPI 把 LLM 包装成 API 服务"的最小部署实验(1 小时)。

### ⚠️ 建议 1 — LangChain→LangGraph 跳跃陡峭

Day 3 只讲 LangChain 基础 Chain,Day 6 直接跳到 LangGraph 状态机。中间的 Runnable/Router/多 Chain 组合被跳过。0 基础可能跟不上。

**改法**: Day 6 开头加 15 分钟"从 Chain 到 Graph 的演进"过渡讲解。

### ⚠️ 建议 2 — FastAPI/React 未覆盖但出现在技术栈

§1.3 提到 FastAPI + React,但 7 天没讲。隐含假设"6 年开发经验 = 会 FastAPI/React"。

**改法**: 明确在文档里写"FastAPI/React 假设你已掌握,不在学习范围"。

## 维度三:遗漏项

### ❌ 问题 3 — 日历映射未明确,难度与时间倒挂

计划没说"Day X = 星期几"。按自然推算(今天周日 6/22):

| Day | 日期 | 可用时间 | 内容 | 难度 |
|---|---|---|---|---|
| Day 4 | 周四 6/26 | **1.5h** | Function Calling ⭐(RCA 发动机) | 最高 |
| Day 5 | 周五 6/27 | 1.5h | Embedding(相对次要) | 中 |
| Day 6 | 周六 6/28 | **8h** | LangGraph ⭐⭐ | 最高 |
| Day 7 | 周日 6/29 | 8h | RAG + 串联 | 高 |

**Day 4(最关键的 Function Calling)只有 1.5 小时,而 Day 5(相对次要)也是 1.5 小时。** 难度与时间严重倒挂。

**不修改的后果**: Day 4 学不透,后面 Day 6/7 的串联全垮——Function Calling 是整个 RCA 的发动机。

**改法**:
1. 明确写出日历映射
2. 把 Day 4 和 Day 6 内容对调周末,或 Day 5/6 互换让 FC 落在周六(8h)
3. 推荐重排:Day 4(周四)= Embedding,Day 5(周五)= Function Calling + 周六加练,Day 6(周六)= LangGraph

### ❌ 问题 4 — 无缓冲日

Day 7(周日)是赛前最后一天,还在学新内容(RAG 串联)。没有排练/复习/补漏的时间。

**改法**: Day 7 拆为"上午学 RAG,下午完整排练 + 查漏"。或加一个 Day 8(周一比赛日早上)2h 快速复习。

### ⚠️ 建议 3 — 无 API 失败/成本应急预案

- 硅基流动 API 挂了怎么办?
- 免费额度用完了怎么办?
- 比赛/演示当天网络不稳?

**改法**: 加一个"应急方案"小节,备选 DeepSeek API(直连,不依赖硅基流动)或本地 Ollama 兜底。

### ⚠️ 建议 4 — 版本未锁定

LangChain/LangGraph 版本迭代极快,API 经常变。未指定版本会导致教程代码跑不通。

**改法**: requirements.txt 里 pin 版本,如 `langgraph>=0.2,<0.3`。

## 维度四:可执行性

### ❌ 问题 5 — 环境搭建时间未预算

Day 1 包含"注册硅基流动 + 获取 API Key"。注册 + 实名认证 + Key 配置很容易吃掉 30+ 分钟,只剩 60 分钟学概念+写代码。

**改法**: Day 0(今晚,周日)花 30 分钟把环境搭好(注册 + Key + pip install),Day 1 直接学。

### ⚠️ 建议 5 — "30 行代码"限制对 Day 4/6 不现实

Function Calling 完整示例(工具定义 + 调用 + 结果回传)约 50-60 行。LangGraph 最小示例约 40 行。30 行限制会导致代码被过度压缩,反而看不懂。

**改法**: 代码量上限放宽到 60 行,或对 Day 4/6 单独标注"允许 60 行"。

### ⚠️ 建议 6 — 验收标准是占位符

§三 的验收清单写的是 `[ ] 能回答 X` ` [ ] 代码跑通`,X 未定义。

**改法**: 评审通过后,生成详细内容时必须填入具体勾选项,不能留占位符。

这是一份**定位清晰、结构合理**的学习计划,RCA 锚定原则和问答驱动形式值得肯定。但存在**三个致命问题**:(1) 理论权重严重不足,而理论是明考项;(2) Function Calling 这个最关键的发动机环节只给了 1.5 小时;(3) 没有比赛排练缓冲日。按现状执行,大概率"项目能搭出来但理论丢分 + 关键环节学不透"。

## 亮点

1. **定位极其清晰** — §1.4 "不是什么/是什么"表格,避免了"大而全"陷阱
2. **RCA 锚定原则** — 每个知识点映射到具体环节,学习不跑偏
3. **问答驱动形式** — 5 个 Q&A/天 模拟比赛理论题,一举两得
4. **诚实的的时间预算** — 明确算出 23.5h,没有自欺欺人
5. **自测 + 验收结构** — 每天有检验机制(前提是填具体项)
6. **优先级分层** — ⭐ 标记让重点一目了然

## 盲区清单(隐含假设)

1. **假设理论能从 5 个 Q&A/天学会** — 低估了理论考试深度,Transformer/训练流程不是 5 个问答能覆盖的
2. **假设 1.5h 够学最难概念** — Function Calling 和 LangGraph 对 0 基础来说,1.5h 只够"看懂",不够"会用"
3. **假设环境/API 搭建是小事** — 注册 + 实名 + Key + pip 报错,轻松吃掉 1 小时
4. **假设比赛实操 = 演示现有项目** — 如果是"现场限时搭建",7 天学习无法转化为限时产出
5. **假设不需要排练日** — 比赛演示需要肌肉记忆,Day 7 还在学新内容 = 没排练
6. **假设硅基流动免费额度够用** — Function Calling + LangGraph 调试会消耗大量 token,免费额度可能不够
